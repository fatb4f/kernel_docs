# pkt-r08-01 — Materialize typed manifest declarations

**Status:** final
**Plane:** policy
**Depends on:** current baseline
**Enables:** pkt-r08-02

## Goal
Turn manifests/ from directory scaffolding into a typed declaration layer for bundle, projection, and generator control objects.

## Why this packet exists
- Later packets need a stable control surface before normalization and admission can be typed.
- The current packet description is too vague: it names areas to touch but does not define what must become true when the packet lands.
- This packet closes the declaration-model gap without activating rendering, generation, or runtime execution.

## Scope
### In scope
- Typed manifest control schema with class discrimination
- Minimal manifest inventory and catalog/index
- One minimal valid control object per manifest class
- Invalid examples proving edge rejection
- Lifecycle distinction between active and declared
- Structured input/output members for control objects

### Non-goals
- Jsonnet rendering
- Generator execution
- Python/runtime code
- Executable admission logic
- Derived output generation

## Design rules
- Do not introduce runtime/controller semantics into kernel manifest declarations.
- Do not let declared manifests imply executable handlers exist.
- Do not use raw string-only outputs in normative examples.
- Do not point manifest outputs into build/.
- Do not let manifest declarations point outside kernel/.

## Required member matrix

### Common required members
| Member | Type | Required | Rule |
|---|---|---|---|
| kind | string | Yes | constant = kernel-manifest-control |
| manifest_id | string | Yes | stable patterned identifier |
| spec_version | string | Yes | constant = v1 |
| manifest_class | string | Yes | enum = bundle|projection|generator |
| status | string | Yes | enum = active|declared|deprecated |
| owner_plane | string | Yes | enum = policy|data_shaping |
| inputs | array<object> | Yes | structured refs only |
| outputs | array<object> | Yes | structured outputs only |
| admission | object | Yes | declares admission requirement and scope ref |
| regen | object | Yes | declares regeneration policy |
| notes | string | No | optional human note |

### Structured input shape
Required members: `ref, input_kind, required`
- ref is a repo-relative or logical contract reference
- input_kind identifies the dependency role
- required is boolean
- raw glob-only entries are not allowed in normative examples

Example:
```json
{
  "ref": "schemas/exported/exported-surface.index.json",
  "input_kind": "contract_index",
  "required": true
}
```

### Structured output shape
Required members: `output_class, path, committed`
- output_class identifies the declared output family
- path is repo-relative and must stay under kernel/
- committed is boolean
- paths into build/ are forbidden

Example:
```json
{
  "output_class": "bundle_index",
  "path": "generated/state/bundles/kernel-core.index.json",
  "committed": true
}
```

### Class-specific requirements
#### bundle
Required members: `bundle_id, bundle_scope, includes, excludes`
- must not include runtime package paths
- must not include build/
- must not include external provider runtime artifacts

#### projection
Required members: `projection_id, input_state_class, renderer_identity, output_class`
- renderer_identity may be declared but not executable in r08.01
- status should be declared for the initial projection manifest

#### generator
Required members: `generator_id, generator_identity, contract_input_refs, derived_output_classes`
- generator_identity may be declared but not executable in r08.01
- status should be declared for the initial generator manifest

### Surface index required members
Required members: `packet_id, active_manifest_root, manifests`

## Lifecycle matrix

## Index rules
- Every manifest file introduced by this packet must appear exactly once in manifests/surface.index.json.
- The surface index must record manifest_id, manifest_class, status, and path for each manifest instance.
- The surface index is authoritative for declared manifest inventory in r08.01.
- No manifest path may point outside kernel/.

## Acceptance criteria
### Schema Acceptance
- manifest-control-family.schema.json uses class discrimination.
- bundle, projection, and generator have separate required-field sets.
- plain string-only outputs are rejected.
- missing manifest_id is rejected.
- invalid manifest_class is rejected.

### Repository Acceptance
- Exactly three manifest files are materialized in this packet.
- Exactly one manifest exists per class in this packet.
- Every manifest is listed in manifests/surface.index.json.
- No manifest path points outside kernel/.
- No manifest output points into build/.

### Boundary Acceptance
- No manifest includes runtime/controller semantics.
- No manifest declares executable behavior.
- Projection and generator manifests remain declared, not active.
- The bundle manifest is the only active manifest in this packet.

### Example Coverage
- Invalid examples fail for invalid class.
- Invalid examples fail for missing manifest id.
- Invalid examples fail for malformed structured output.
- Invalid examples fail for runtime/controller leakage.

## Implementation order
1. Expand manifest-control-family.schema.json into a discriminated union.
2. Create the three concrete manifest control objects.
3. Update manifests/surface.index.json.
4. Add valid examples for each class.
5. Add invalid examples for schema and boundary rejection.
6. Update valid/invalid example indexes.
7. Update README.md and kernel.spec.md to match the final declaration model.
