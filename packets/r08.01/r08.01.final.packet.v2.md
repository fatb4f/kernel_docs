# pkt-r08-01 — Materialize typed manifest declarations

**Revision:** r2  
**Status:** reconciled-final  
**Plane:** policy  
**Depends on:** current baseline  
**Enables:** pkt-r08-02

## Goal
Turn manifests/ from directory scaffolding into a typed declaration layer for bundle, projection, and generator control objects, with a single canonical manifest contract, a single path basis, and an explicit example migration.

## Scope
### In scope
- Typed manifest control schema with class discrimination
- Minimal three-manifest declaration inventory and authoritative surface index
- One valid example per manifest class and explicit invalid examples
- Lifecycle distinction between active and declared
- Canonical repo-relative path rules
- Explicit migration from legacy manifest prose fields and legacy example names

### Non-goals
- Jsonnet rendering
- Generator execution
- Python/runtime code
- Executable admission logic
- Derived output generation

## Canonical contract rules
This packet JSON and its matching packet markdown are the only normative definition of the r08.01 manifest contract. Any companion prose that uses legacy field names must be updated in the same change or marked obsolete and removed from normative references.

### Legacy field translation
| Legacy field | Canonical field | Rule |
|---|---|---|
| `id` | `manifest_id` | Do not use both. manifest_id is authoritative. |
| `input_refs` | `inputs` | Translate to structured input objects with ref, input_kind, and required. |
| `output_class_ids` | `outputs` | Translate to structured output objects with output_class, path, and committed. |
| `consuming_plane` | `owner_plane` | owner_plane is authoritative. |
| `admissibility_requirement` | `admission` | Translate to an admission object with required and scope_ref. |
| `handler_id` | `renderer_identity or generator_identity` | handler_id is not allowed in the canonical top-level contract. Bundle manifests have no handler identity. |
| `regen_policy` | `regen` | Translate to an object. regen is authoritative. |

### Forbidden legacy top-level fields
- `id`
- `input_refs`
- `output_class_ids`
- `consuming_plane`
- `admissibility_requirement`
- `handler_id`
- `regen_policy`

### Path basis
- All manifest and index path values in this packet are repo-relative and must begin with kernel/.
- Allowed manifest root: `kernel/manifests/`
- Allowed output root: `kernel/generated/`
- Disallowed examples:
  - `manifests/bundles/kernel-core.bundle.json`
  - `generated/state/bundles/kernel-core.index.json`

Legacy family-level example names are replaced by class-specific example names in this packet. The example indexes must be migrated in the same change so there is one authoritative example registry.

## Index rules
- Every manifest file introduced by this packet must appear exactly once in kernel/manifests/surface.index.json.
- Every path stored in kernel/manifests/surface.index.json must be repo-relative and begin with kernel/.
- The surface index is authoritative for manifest inventory in r08.01.
- Valid and invalid example indexes must remove legacy family-level entries and list only the class-specific examples introduced by this packet.

## Acceptance criteria
### Schema Acceptance
- manifest-control-family.schema.json uses class discrimination.
- bundle, projection, and generator have separate required-field sets.
- plain string-only outputs are rejected.
- missing manifest_id is rejected.
- invalid manifest_class is rejected.
- legacy top-level fields such as id, input_refs, output_class_ids, consuming_plane, admissibility_requirement, handler_id, and regen_policy are rejected by canonical examples and schema tests.

### Path Acceptance
- No manifest or index path in this packet drops the kernel/ prefix.
- No structured output path points outside kernel/generated/.
- No manifest path points outside kernel/.

### Packet Set Consistency
- No packet prose defines an alternative top-level manifest contract.
- Any companion manifests.md is updated to the canonical field names or explicitly removed from normative use.
- Markdown and JSON packet artifacts match on field names, path rules, and example migration rules.

### Example Migration Acceptance
- kernel/examples/valid/example.index.json removes legacy family-level manifest-control-family entries.
- kernel/examples/invalid/example.index.json removes the legacy manifest-control-family.invalid.example.json entry.
- The valid example index lists exactly the three class-specific valid examples introduced by this packet.
- The invalid example index lists exactly the four targeted invalid examples introduced by this packet.

### Boundary Acceptance
- No manifest includes runtime/controller semantics.
- No manifest declares executable behavior.
- Projection and generator manifests remain declared, not active.
- The bundle manifest is the only active manifest in this packet.

## Implementation order
1. Expand manifest-control-family.schema.json into a discriminated union using the canonical field set.
2. Update or retire any packet-set prose that still uses the legacy field set.
3. Create the three concrete manifest control objects.
4. Update kernel/manifests/surface.index.json using canonical kernel/ paths.
5. Replace legacy example index entries with the class-specific valid and invalid examples.
6. Update kernel/README.md and kernel/kernel.spec.md to match the canonical declaration model.
